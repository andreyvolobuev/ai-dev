# Книга правил — Backend Dev (bellingshausen)

Этот файл подкладывается в system prompt backend-dev-агента, работающего с репо `bellingshausen`.

**Формат:** каждое правило — одна строка, желательно с кратким примером или обоснованием.

Агент может предлагать дополнения в этот файл после каждой задачи; одобряет их человек.

## Стиль кода
- Используем двойные кавычки для строк (проект на ruff с `quote-style = "double"`).
- Импорты сортируются через ruff (isort).

## Комментарии
- **Комментарий объясняет ЗАЧЕМ, а не ЧТО.** Имя функции / переменной уже говорит что делается.
- По умолчанию — не писать. Писать только если:
  * есть неочевидный инвариант или ограничение (например, "эта проверка нужна, потому что API возвращает None при пустом результате"),
  * это workaround известного бага (с указанием тикета/ссылки),
  * поведение противоречит интуиции и без комментария читатель сделает неверное допущение.
- Не писать «служебные» комментарии типа `# проверяем версию pydantic` перед `if pydantic.VERSION > ...` — это шум.
- Не ссылаться на текущий тикет или "added for X" в комментариях кода — это в MR описание.
